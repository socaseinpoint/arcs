§ 01

What it is, in one breath.

A tiny file method — two primitives, state is a folder.

arcs is a tiny file-based method: every task is an input → workspace → output folder living under a hidden .arcs/ at your project root. The agent runs it; you read the results. No new app, no database — just folders, named so finished work is obvious at a glance. The method is tool-independent; the CLI and the agent skill only serve it.

§ 02

Why it matters.

Three payoffs — one line each.
// where did we stop?

Status persists across sessions. A fresh agent — or you tomorrow — asks "where did we stop?" and gets the answer from the files, or one arcs status. A cold handoff that just resumes, no chat-scrolling.

// ideas surface, don't leak

An idea that pops up mid-task isn't lost in the scroll. arcs candidate parks it, and it bubbles up onto the board until you promote it. Future work surfaces instead of evaporating.

// agent-native

The agent structures and parallelizes the work itself — each arc is an encapsulated unit, so independent arcs run as parallel subagents. Goals are immutable and chain, so every pivot stays traceable.

§ 03

Install once. Then just talk to your agent.

You don't run arcs by hand — your agent does. You install the CLI one time, then work through Claude Code as usual.
# 1 · install once — clone anywhere, run the installer
git clone https://github.com/socaseinpoint/arcs ~/arcs && cd ~/arcs && ./install.sh

One command. It adds arcs to your PATH, links the Claude skill, and registers the enforcement hooks — so every session in an opted-in project nudges (and gates) the agent toward arcs. Open a new shell afterwards. Setup for your OS →

2 · opt your project in — once. Run arcs init in the project directory and from then on the work binds to that directory: it lands in a hidden .arcs/ at the root, so your code is never cluttered, and the thread lives with the repo instead of inside a chat. One command you remember; everything after it is the agent's job.

That's the whole record — legible to you at a glance and to the agent that keeps it. Plain named folders, no app, no database. Finished work is obvious from a bare ls.

3 · then just tell your agent what to build. Open your project in Claude Code and describe the work. The agent drives arcs for you — it opens an arc or goal, works input → workspace → output, and closes it when the result is self-contained. You never type the commands; the structure just appears on disk as the agent goes.

peek if you want. The one line a human might actually run — arcs status — prints the board the agent is keeping: what's open, what's done, where the work stands, plus any parked candidates. Everything else (new-goal, new-arc, close…) is what the agent runs, not a how-to for you. The full command set and per-OS setup live in the docs.

§ 04

Works with an agent — or without.

The method stands on its own: folders plus the encapsulation rule.

Without an agent, arcs is a discipline — you create the folders and keep output/ self-contained yourself. The value (structure, and a "you are here" you can re-open any time) holds even solo.

With an agent, arcs enforces itself: a skill advises, a hook reminds, and a gate hard-blocks edits until an arc is open — so the method turns from "must remember" into "can't skip." That's where the payoff peaks: a cold handoff between sessions and agents that just resumes. Needs only bash + grep/sed, no build step. See the docs for the full model.

Create .arcs/ → let the agent work inside arcs → never lose the thread.

Open source. The CLI and skill live outside your project; a project gets only .arcs/.
