§ documentation

The arcs method, in full.

The landing tells you the value in thirty seconds. This page is the mental model: what arcs is, where the state lives, how the lifecycle runs, and why each rule exists. Every command and field below is the real CLI behavior.

01 What & why

Agent work has a failure mode: it happens in an ephemeral chat. Decisions, dead-ends and half-finished artifacts scroll away. The next session — or the next agent — starts cold, with no source of truth for "what was actually done."

arcs answers that with one move: the files are the record. Every chunk of work is a folder with a fixed shape — input → workspace → output — under a hidden .arcs/ at the project root, so the method never clutters your code. State lives on disk, not in a conversation. The method is independent of tooling: agents, chats and IDEs are implementation details. Turn off every optional rule and arcs still runs as plain, named folders.

the one rule that makes it work

Encapsulation. Anything the next step needs must be derivable from output/ alone. workspace/ is private and never leaks; input/ is what came in. Get this right and any agent, any session, can pick up the work from its output/ with zero chat history.

02 Concepts

Everything lives in one place and follows two primitives. There is no separate goals/ directory — arcs and goals share a single numbered stream.

The single stream

All work sits in .arcs/arcs/ under one continuous numbering sequence. Arcs and goals draw from the same counter, so an arc and a goal never share a number. Closing renames an entry but never frees its number.

arc vs goal

arc

work without a purpose

A folder NN-<slug>/ — one finished chunk. Need an answer but don't want it lost in the chat → drop an arc. One-shot, self-contained. It carries input/, workspace/, output/, and a small arc.md passport.

goal

an arc with a purpose

A folder NN-@<slug>/ — the @ marks it. The same skeleton, plus its own nested arcs/ substream and an immutable <slug>-goal.md status doc. For a project or deep research — meaningful, multi-arc work.

input → workspace → output

Each arc has the same three folders. The OOP analogy is exact:

Folder / fileRoleOOP analogue
input/what entered the work context — seed, requirements, ancestor artifactsconstructor args
workspace/all work in progress — drafts, dead-ends, scratchprivate state
output/self-contained result — the outside reads ONLY thispublic interface
arc.mdpassport: goal: · status: · pointers to key output—

Slugs must be descriptive — multi-word and content-rich (extract-auth-from-monolith), never generic (fix, tmp). The name is the at-a-glance index of the stream.

03 Lifecycle

One rhythm, the same for arcs and goals:

You open before working: arcs new-arc <slug> (or new-goal) scaffolds input/ workspace/ output/ and a templated arc.md. As you work, notes flow into workspace/ — not dumped at the end. When the result is ready you write it to output/ and close.

Closing marks done two ways at once. arcs close <slug> sets status: done in the status doc and renames the folder NN-<slug> → __NN-<slug>__ (two underscores each side) — so finished work is visible in a bare ls. It refuses an empty output/: a closed arc must carry a self-contained result. Override deliberately with arcs close -f <slug>. arcs reopen <slug> undoes both — strips the __…__, status back to active.

04 Goals — the immutable contract

A goal is the most important idea in arcs. Its intent does not get edited in place. The doc is a goal: line plus a short "where we are" — one <slug>-goal.md per goal. Two rules follow from immutability.

1 · Immutable intent → supersession chains

When the aim changes — a real change of meaning, not a wording fix — you don't rewrite the goal. You close the old one and open a new one that carries supersedes: <old-slug>, leaving a chain you can trace. The CLI does it in one move:

Both stay on disk, linked — a pivot you can trace, not a rename. References are bare folder slugs (the parser tolerates __…__). prev: is accepted as a read-only alias of supersedes:. Clarifying edits are still fine in place; supersede only when the meaning moves.

2 · The checklist is the sub-arcs

A goal has no separate ## Checklist block. Its items are its sub-arcs: planning a goal means opening its sub-arcs up front (each an open NN-<slug>/), and an item is done when its sub-arc is closed (__…__). Nothing is hand-ticked — done-state is the folder state, read straight from disk. That kills a whole class of drift a hand-authored item list invites (a mistyped key, a dotted key that won't match).

N/M ✓ (closed sub-arcs / total) on the goal line, then ✓ for each closed sub-arc and ○ for each open one; a superseded sub-arc shows its supersedes link. The goal doc answers, briefly: the goal and where you are globally — re-open it after a deep dive and you instantly see "you are here."

05 Rules layer

The base method — arc · goal · encapsulation · versioned status — is complete on its own. Rules are an opt-in layer on top: per-project behaviors an agent follows while working. Turn them all off and the method runs exactly as specified.

Rule bodies are global. They ship in the method repo at <arcs-repo>/rules/<slug>.md, versioned with the method — so arcs update brings new and updated rules to every project at once. The switch is per project, in .arcs/config: one comma-separated rules= line. Drop .arcs/rules/<slug>.md to override a global rule of the same name or to add a project-only one — local wins. Toggling never edits a rule file, only the rules= line.

the built-in: subagents

arcs init writes rules=subagents by default. Its body lives globally at <repo>/rules/subagents.md. The idea: an arc is already encapsulated (input → workspace → output, outside reads only output/), so the arc boundary IS a subagent boundary. With the rule on, an agent dispatches one subagent per arc — independent arcs in parallel, dependent ones pipelined — while the orchestrator reads back only each arc's output/. Off, the method still works: run the arcs inline, same files.

06 Candidates — the backlog

An idea that surfaces mid-work — a candidate for a future arc — must not be buried in the current arc's workspace/. Workspace is private and disposable: it dies with the arc. So a future-work idea has to be surfaced out onto the board, not left to rot.

.arcs/candidates/ sits beside .arcs/arcs/. It holds surfaced candidates as numbered NN-<slug>.md files — their own numbering sequence, separate from the work stream. Each carries a from: field (the arc it surfaced from — a traceable origin) plus a free-form body.

arcs status shows the backlog in a CANDIDATES section. Promote only when you actually start the work — the candidate's body becomes the new arc's seed.

07 Command reference

The full set, exactly as bin/arcs dispatches them. In normal use the agent runs these — you describe the work in Claude Code and it opens, works, and closes arcs for you; arcs status is the one a human might run to peek at the board. Field keys (goal:, status:, supersedes:) stay English so the tooling is language-agnostic.

CommandWhat it does
arcs init [en|ru|es]create .arcs/ + pick the arc language (asks if a TTY); writes rules=subagents.
arcs lang [en|ru|es]show or change this project's arc language.
arcs new-arc <slug>new standalone arc → .arcs/arcs/NN-<slug>/.
arcs new-arc -g <goal> <slug>new arc inside a goal's nested arcs/ (local 01,02…).
arcs new-goal <slug>new goal → NN-@<slug>/ in the shared stream, with its own substream + <slug>-goal.md.
arcs close [-g <goal>] [-f] <slug>mark done: status: done + rename to __NN-slug__. Refuses an empty output/; -f overrides.
arcs reopen [-g <goal>] <slug>undo a close: __NN-slug__ → NN-slug, status back to active.
arcs supersede [-g <goal>] <old> <new>close <old> (no output guard) + create <new> linked by supersedes: + seed input/from-<old>.md.
arcs candidate <slug> [--from <arc>] [text]capture a surfaced arc-candidate in .arcs/candidates/.
arcs candidate listlist candidates (also shown by arcs status).
arcs promote [-g <goal>] <NN-or-slug> [<new-slug>]turn a candidate into a real arc — moves the file into the new arc's input/.
arcs rule [list|on|off|add] <slug>toggle behavior rules (global bodies, .arcs/config switch).
arcs statusprint the board — one STREAM of arcs + goals, plus CANDIDATES.
arcs updategit pull the method + re-wire skill/hooks.
arcs helpthe command list.

08 Setup & install

The usage model first: you install the CLI once; from then on your agent drives it. In day-to-day work you open your project in Claude Code, describe what to build, and the agent runs arcs for you — opening arcs and goals, working input → workspace → output, closing them. A human rarely types arcs at all (arcs status to peek at the board is the one exception).

The tooling lives in the repo — clone it anywhere (~/arcs, ~/code/arcs, whatever). A project only ever gets a hidden .arcs/ dir; the method never clutters your code. The CLI is bash, so each OS below is just "get a bash shell, clone, run the installer."

macOS / Linux

Native bash — the one-command path. install.sh picks your shell's rc file from $SHELL (~/.zshrc for zsh, ~/.bashrc for bash, else ~/.profile) and appends a PATH line; if ~/.claude/skills exists it symlinks the Claude skill to ~/.claude/skills/arcs; and if ~/.claude + python3 are present it registers the enforcement hooks in ~/.claude/settings.json (with a .bak backup, idempotent). Open a new shell afterwards.

Prefer to wire PATH by hand? Clone wherever you like and point PATH at that clone's bin/:

Windows

arcs is a bash CLI, so on Windows you run it inside a bash environment — either WSL (recommended) or Git Bash. Either way the steps are the same shape as macOS/Linux; the difference is where "home" lives, which is where PATH, the skill symlink, and the hooks land. Native PowerShell / cmd is not supported yet — there's no PowerShell port of the CLI.

recommended

WSL (Ubuntu)

A real Linux home, the closest match to how arcs is built.

lighter

Git Bash

Ships with Git for Windows — no Linux subsystem, just a bash shell over your Windows files.

Inside WSL, "home" is the Linux home (/home/<you>). PATH lands in that home's ~/.bashrc, and the skill symlink + hooks land under that home's ~/.claude/ — so run your WSL Claude Code session from there. Clone the project inside WSL too (or work on a path WSL can reach) so the agent and the CLI share one filesystem view.

Git Bash's $SHELL is bash, so PATH appends to ~/.bashrc in your Git Bash home (C:\Users\<you>). The skill symlink + hooks register under ~/.claude/ there, provided python3 is on PATH for the hook step — without it the installer skips hooks and prints the manual line to add them. If arcs help doesn't resolve in a fresh Git Bash, re-open the shell (or source ~/.bashrc).

Enforcement (recommended)

Three layers, weakest to strongest — install.sh registers the hooks automatically (in ~/.claude/settings.json, with a .bak backup, idempotent):

  1. skill — advises the agent to use arcs.
  2. arcs-hook (SessionStart + UserPromptSubmit) — injects a reminder + the live board each session/prompt in an opted-in project, so the agent can't quietly forget.
  3. arcs-gate (PreToolUse Edit|Write|MultiEdit) — a hard block: refuses edits to project files while .arcs/ has no active arc. Edits inside .arcs/ are always allowed; projects without .arcs/ are never affected.

Opting a project in

One command, run once per project — and in practice the agent runs it the first time you point it at fresh work; you don't have to. After that, the agent opens arcs and goals on its own:

Updating

One command, from anywhere — the CLI knows where it was cloned. It git-pulls the method and re-runs the installer, so PATH, skill, and hooks are re-wired idempotently. Then restart your Claude Code session so it reloads the skill + hooks. Your projects' .arcs/ data is untouched.

Requirements: bash + standard coreutils (grep, sed); python3 for hook registration. macOS / Linux natively; Windows via WSL or Git Bash. No build step. .gitignore: .arcs/ is usually committed — it's the memory of the work. If you'd rather not, add .arcs/ to the project's .gitignore.

09 Worked example

A tiny project from examples/basic/: one stream holding a throwaway arc and a goal whose checklist is its sub-arcs. Watch one sub-arc close and the board compute 1/3 ✓ from disk.

The checklist is just the sub-arcs; __01-research__ is closed, so the board ticks it — progress computed from disk, never hand-marked:

End to end: arcs init → arcs new-goal newsletter → open the items as sub-arcs: arcs new-arc -g newsletter research, … draft, … ship → work the first, write output/shortlist.md → arcs close -g newsletter research. The next arcs status shows 1/3 ✓ — read straight from disk. The drafting arc reads only __01-research__/output/shortlist.md — never its workspace.
